# WorkLog Daily Report Skill

[English](README.md) | 简体中文

一个用于 Codex 的 skill：帮你在还记得的时候先把工作过程记下来，再整理成可确认的清洗稿，最后生成日报、周报/月报素材和述职复盘材料。

## 问题：不是不会写，是过后想不起来

写日报、周报、月报、述职材料时，最难的往往不是“怎么写得漂亮”，而是事情过去以后，已经想不清楚了。

常见痛点是：

- 不知道自己到底干了什么。到下班、周五或月底才开始回忆，只能从聊天记录、会议、commit、工单和零散印象里倒推。
- 细节记不清。比如一项工作最开始的思路是什么，为什么改方案，中间试过哪些办法，谁给过输入，最后是怎么一步步迭代出来的。
- 总结时体现不出工作量。结果可能只是一句话，但背后有调研、沟通、排查、返工、判断和取舍；如果当时没留下过程证据，述职和工作总结里就显得很薄。

这个 skill 适合那些你本来就能顺手记录几句的场景：

- 地铁上 5 分钟；
- 等外卖的时候；
- 开完会走回工位的路上；
- 下班前快速复盘；
- 刚和别人聊完，有一些新判断的时候。

你只需要先把 raw text 扔出来。它的目标不是把普通工作包装得很夸张，而是保留真实过程：你做了什么、为什么这么做、怎么迭代、投入了哪些隐性工作，以及这些内容以后如何变成日报、周报、月报或述职证据。

## 适合谁？

适合：

- 试用期员工、实习生、新入职员工；
- 需要写日报、周报、月报的人；
- 准备晋升、转正、述职材料的人；
- 研究员、工程师、量化、分析师、产品经理；
- 所有习惯“先碎碎念，再整理思路”的人。

## 解决方案

工作流非常简单：

```text
raw text notes
  -> cleaned review draft
  -> 用户确认
  -> daily worklog / 周报素材 / 述职证据
```

这个 skill **不负责语音转文字**。

你可以用任何自己喜欢的输入方式。中文场景里，一个很实用的流程是：

1. 用豆包输入法、系统听写或其他语音转文字工具。
2. 对着手机/电脑碎碎念几分钟。
3. 把转出来的 raw text 贴给 Codex。
4. skill 先整理成 cleaned review draft。
5. 你快速 review 一遍，确认意思没被改歪。
6. 明确批准。
7. 再生成日报、周报素材或述职材料。

## 为什么一定要先 review？

这个 skill 不会直接从 raw notes 跳到最终报告。

它会先生成一版 **cleaned review draft**。

这版稿子仍然接近原始意思，但会做轻度结构化，方便你快速 review：

- 长段落会被自动拆分；
- 重复和口语化废话会被去掉；
- 相关内容会被归类；
- 明显语音识别错误会被修正；
- 同事、项目、团队、公司、工具等名称会尽量和历史记录保持一致；
- 不确定的修正会标出来，而不是偷偷改写；
- 工作证据、social / 环境信息和非工作内容会被分开；
- 不确定内容和专用名词修正会标出来，方便你确认。

只有你明确确认后，skill 才会写最终日报。

这一步的核心是保护你的原意：**总结可以变清楚，但不能把意思改歪。**

## 设计原则

### 1. raw notes 是证据

raw notes 应该被保存下来，作为原始证据。skill 不应该直接美化或覆盖 raw。

### 2. 先清洗，再写报告

日报应该建立在你 review 过的清洗稿上，而不是模型直接对一团乱输入做最终解释。

### 3. 保留原意优先于文笔优美

skill 可以提升可读性，但不能编造事实、日期、指标、决策、阻塞和结果。

### 4. 适应职场场景

默认输出面向工作复盘：

- 做了什么；
- 学到了什么；
- 和谁沟通过；
- 做了什么判断；
- 遇到了什么问题；
- 下一步是什么；
- 哪些内容以后能用于周报、月报、转正或述职。

### 5. 非工作内容也可以保留，但要分类

真实碎碎念里经常有生活、情绪、想法和反思。skill 可以保留它们，但应该清楚标注，避免混进正式工作报告。

### 6. 保持长期一致性

工作日志是长期积累的。一个同事、项目、公司、工具或反复出现的概念，如果之前已经出现过，skill 应该利用已有上下文修正常见识别错误，保持命名一致。

例如：之前记录里已经出现过某位同事或某个项目名，而新的 raw text 里出现了一个明显像语音识别错的版本，cleaned draft 可以统一成历史上更可信的写法，并在必要时标注这次修正。如果不确定，就应该标出来，而不是直接猜。

这是 review 阶段的职责，不是无控制自动化。skill 会优先利用已确认日报、历史上下文或已有 glossary-like 信息做轻量对齐；它不要求、也不承诺维护一套持久 glossary / context memory 系统。

### 7. 根据目标给 comment

在日报确认生成之后，如果用户有明确目标，例如转正、季度述职、年报或晋升准备，skill 可以给一段简短的 goal-relative comment。

这段 comment 应该区分 evidence、assessment、risk / gap 和 suggested next action。它不能推断“能不能转正”、manager 期待或没有证据支持的私人信息。如果没有目标，就应该省略，或者明确标注 `Goal: not provided`。

## 自动化边界

这个 skill 可以帮助生成日报、周报、月报和述职材料，但不应该静默发布或覆盖最终报告。

推荐职责边界：

- skill 负责清洗 raw notes 和起草报告；
- 用户负责确认 cleaned notes 和重要最终输出；
- 日报可以在确认后生成；
- 周报、月报、述职材料应该基于已经确认过的日报汇总，而不是直接从未 review 的 raw notes 自动生成。

## 安装

把 skill 文件夹复制到 Codex skills 目录：

```text
worklog-daily-report/
```

项目内安装：

```text
.codex/skills/worklog-daily-report/
```

Windows 用户全局安装：

```text
%USERPROFILE%/.codex/skills/worklog-daily-report/
```

然后对 Codex 说：

```text
帮我把这些 raw notes 整理成 daily worklog。
```

它应该先给你一版 cleaned review draft，并等待你确认。

## 开源安全提醒

skill 本身是通用的，但不要把你的私人 worklog 数据一起开源。

避免提交：

- 公司名称；
- 老板/同事姓名；
- 保密项目细节；
- 薪酬信息；
- 交易/策略细节；
- 个人绩效记录。
